管理責任: 実装前にテストを作成し、意図した理由でREDになることを確認します。
ワークフロー: build(test) → run(red)
状態確認先: `tmp/{work-dir}/phase-state.json`の`test`

権限: `source-write`
副作用: テストコード作成、`tmp/{work-dir}/`更新
承認: 不要です(マイルストーン/Issue・設計の範囲内で自走します)。

テストを先に作成し、期待通り失敗(RED)することを確認します。担当Role: Programmer。

## 読み込む文書

1. 対象の設計(`docs/milestones/m{N}-*.md`の実装計画、またはProduction Modeの設計提案)を
   読みます
2. 対象モジュールのディレクトリのREADME.md(概要/責任/責任外、対応テストの記載)を読みます
3. `tests/unit/`配下の同種テストを確認し、既存の書き方の慣習(bats + `nix eval --impure`等)
   に合わせます

## 手順

1. 設計のテスト方針に従い、対応するテスト層(`tests/unit/`配下、対象ディレクトリと
   ミラーする場所)へテストを作成します
2. テスト作成が適さない作業(文書のみの変更等)の場合は、スキップ理由を`phase-state.json`
   の`skip_reasons`へ記録し、実装へ進みます
3. テストを実行し、**期待した理由で失敗する(RED)**ことを確認します
   - 失敗しない場合: テストが対象を検証できていないため、書き直します
   - 想定外の理由で失敗する場合: 前提の崩れを調査し、必要ならPMへ報告します
4. `just verify`を実行します。REDを確認するために意図して失敗する検証がある場合は、
   失敗理由と対象テストを記録します
5. `phase-state.json`の`test`を完了へ更新します
6. commitは行わず、検証・レビュー後に出荷フェーズでまとめます
