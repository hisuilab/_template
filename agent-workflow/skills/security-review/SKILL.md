権限: `tmp-write`
副作用: `tmp/{work-dir}/security-review.md`の作成・上書き
承認: 不要です。

セキュリティ評価軸と手順を定義します。

## 評価軸

### 1. 秘密情報・認証情報の露出

- 秘密情報、トークン、個人環境の値がファイルに含まれていないか
- `.env`や認証情報ファイルがコミット対象に含まれていないか(`.gitleaks.toml`で検出できるか)

### 2. 権限ヘッダ整合

- コマンド・agent・skillファイルの権限ヘッダ(管理責任/ワークフロー/状態確認先/権限/副作用/承認)
  が実際の動作と一致しているか
- より高い権限が必要な操作を低い権限で宣言していないか

### 3. 外部公開ゲート

- commit / push / PR作成に承認ゲートが設定されているか(`/ship:pr`のRigor Mode分岐)
- `command-permissions.md`の権限レベルと実際の操作が対応しているか

### 4. 情報漏洩パス

- 生成物・ログ・tmpファイルから機密情報が漏洩しないか
- エラーメッセージに内部構造の詳細が含まれていないか

### 5. サプライチェーン

- 新規依存の追加前に既存ツールで代替できないか確認されているか(`AGENTS.md`の方針)
- flake入力・CI設定にサプライチェーン上の危険がないか(バージョン固定の有無)

### 6. 監査可能性

- 操作履歴と判断理由を追跡できるか(`docs/decisions/`との連携)
- `phase-state.json`で作業状態が記録されているか

## チェック手順

1. 対象差分・設定ファイル・CI設定・権限関連文書を確認します
2. 評価軸1〜6を順番に確認し、指摘を分類します
3. Critical / High指摘は修正またはPM判断まで出荷を止めます

## 出力形式

`tmp/{work-dir}/security-review.md`へ以下の形式で記録します。

```markdown
## セキュリティレビュー: {work-dir}

### Critical

- ...

### High

- ...

### Medium

- ...

### Low

- ...

### 判定: 承認 / 要修正 / PM判断が必要
```
