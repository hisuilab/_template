## 1. 作業開始

- 対象ディレクトリのREADME.md(概要/責任/責任外)を読み、担当範囲を確認します
  ([`command-permissions.md`](command-permissions.md)、各コマンドの「読み込む文書」節に従います)
- `docs/milestones/README.md`で現在のマイルストーン状況を確認します
- 未知の既存変更はユーザー所有として扱い、上書きしません

## 2. 変更

- 1つの変更理由にScopeを限定します
- entrypointへ検証・計画・実行ロジックを置きません
- 新しい抽象化にはcaller、contract、テストを要求します
- 現行実装と未実装の目標設計を混同しません(目標設計には明示ラベルを付けます)
- Markdownテーブルはコンパクト形式(セル幅のパディングなし)で記述します
- コードブロックには必ず言語識別子を付けます(不明な場合は`text`)
- Mermaid 図には必ず `%%{init: {"theme": "dark"}}%%` を先頭行に置きます
- Mermaid ノードラベルの改行は `\n` でなく `<br/>` を使います
- Mermaid ノードラベル内に Unicode 矢印（`→` 等）を含めません（VS Code パーサーエラーの原因）
- 1マイルストーン/Issue = 1目的、1コミット/PR = 1論点にします。設計変更と実装修正を混ぜません
- 事前承認なしの差分上限は8ファイル・500行・新規5ファイルです
  ([`workflows.md 7.`](workflows.md#7-ループガードレール))。超える見込みが出た時点で分割案を提示します
- 大量rename、整形のみの変更、依存更新、生成物の一括更新は単独のcommit/PRへ分離します

## 3. Gitと外部操作

- commit・push・PR・Issue・Releaseは、[`command-permissions.md`](command-permissions.md)の
  権限レベルと承認ルールに従います。`/build:test`・`/build:implement`単体ではcommitしません。
  `/plan:requirements`・`/plan:architecture`が`project-direction.md`の差分を、`/plan:design`
  が設計文書(チェック前状態)をそれぞれ完了時にcommitし、`/build:docs`が残りの未commit差分
  (test・実装本体・文書同期分)を種別ごとに複数commitへ分割して、1回の承認でまとめて
  連続実行します
- Prototype Modeでは`m{N}-slug`ブランチへ`/plan:design`と`/build:docs`が段階的にcommitし、
  マイルストーン完了後に`/ship:pr`が`prototype`へfast-forward mergeします。`main`への統合
  (Production Mode)はPR経由(squash merge)を必須とします
- `reset --hard`、force push、未追跡ファイル削除などの破壊操作は明示承認なしで実行しません
- 命名は[`naming-policy.md`](naming-policy.md)に従います(commit messageは`AGENTS.md`の
  Conventional Commits形式が正本)

## 4. 完了報告

- 検証は項目ごとに pass / fail / not-run で報告します。実行結果だけを根拠にし、自己評価で
  代替しません
- 変更ファイルと変更理由、残存リスク、人間判断が必要な点、次のアクションを含めます
- 出力の末尾に現在フェーズと次のステップを提示します
- 差し戻し・手戻り・検証すり抜けにつながった失敗は、将来の`docs/ai/lessons.md`相当へ
  記録します(未整備の間は完了報告内に明記するだけで足ります)

## 5. PM-エンジニアモデル

PM(ユーザー)とエンジニア(AIエージェント)が役割を分担します。

| 担当 | 責任 |
| --- | --- |
| PM | ビジョン・仕様の意思決定、各フェーズの承認、外部公開の最終判断 |
| エンジニア | 調査・設計・実装・テスト・レビュー・commitの自律実行 |

### 5.1. 意思決定レポート

PMへの確認は、1タップで返せる次の形式に統一します。

```text
📋 意思決定 — [フェーズ: コンテキスト]

[状況を2行以内で説明]

1. [A案]
2. [B案]
3. [C案] ← 推奨

どれにしますか?
```

- 選択肢は最大3択とし、推奨案を明示します
- 状況説明は2行以内に収めます(長文禁止)
- PMは番号1つを返すだけでエンジニアが再開できるようにします

### 5.2. 確認するトリガー

| トリガー | 例 |
| --- | --- |
| 仕様の解釈が複数ある | どちらの挙動が正しいか |
| 範囲判断が必要 | この機能を本マイルストーン/Issueに含めるか |
| 外部公開操作 | PR作成・pushの実行前(Production Mode) |
| フェーズ完了の承認 | 要件・設計・実装完了の確認 |
| レビューループが上限に達した | 方針変更が必要か |

確認しない(エンジニアが自律判断): ファイル編集、テスト実行、ドキュメント生成、レビューループの進行。

確認する: commit messageと対象差分(Prototype Modeも含む)、push、PR作成、Issue作成、Release。

意思決定の問い合わせはトップレベルのコマンドだけが行います。subagentは確認が必要な事実を
報告に含めてトップレベルへ委譲します。

### 5.3. 承認記録

外部公開操作、`git-commit`以上の操作、HighリスクIssueの例外的な並列実行では、承認を
後から検証できる形で記録します。

| 項目 | 必須内容 |
| --- | --- |
| `approved_by` | 承認者。通常は`PM` |
| `approved_at` | ISO 8601形式の日時 |
| `approval_ref` | GitHub Issue / PRコメントURL、または日時付きチャット引用 |
| `approved_scope` | 承認した操作の範囲 |
| `risk_acceptance` | Highリスクや破壊的操作を例外許可する場合の理由 |

`approved_by: "PM(chat)"`のような自由記述だけでは承認済みとして扱いません。
